## 1. Mục đích

FRD trả lời **"Hệ thống phải làm những chức năng gì?"** — chuyển User Story (URD) thành Functional Requirement có thể test. Đính kèm **Product Backlog** đã ưu tiên theo 4 phase của Feature Tree v4.0.

Quy ước phase: **P1** = Phase 1 (4 tháng) · **P15** = Phase 1.5 stabilization (2 tháng) · **P2** = Phase 2 (6 tháng) · **P3** = Phase 3 (Year 2).

**K-rule áp dụng FR-IND-* (theo ADR-0026):**
- K-1 RLS: 5 customer-facing tables enforce RLS (`customer_workflow_versions`, `workflow_customizations`, `enterprise_industry_bootstrap`, `enterprise_workflow_mode`, `workflow_change_requests`). 6 industry catalog tables platform-shared, no RLS (đúng design cùng node_catalog mig 068).
- K-2: `customer_workflow_versions` immutable trigger refuse UPDATE/DELETE — rollback = new version, không destructive.
- K-12 anti-IDOR: tất cả enterprise-scoped endpoint enforce path `enterprise_id` == `X-Enterprise-ID` header, raise 403 mismatch.
- K-13 idempotency: bootstrap UNIQUE(enterprise_id) → 409 conflict; force=true destructive override.
- K-19 OTel span: `tenant_id` attribute trên every endpoint.

**Acceptance criteria mẫu FR-IND-05 (bootstrap-from-industry):**
> *Given* enterprise X chưa bootstrap, admin call `POST /enterprises/X/bootstrap-from-industry` với body `{industry_id: "retail", dry_run: false}`,
> *When* request có header `X-Enterprise-ID: X`,
> *Then* response 201 với `bootstrap_id` + dept_count=6 + workflow_link_count=15; second call cùng body return 409 `bootstrap_already_done`; second call với `force=true` destructive re-bootstrap.
