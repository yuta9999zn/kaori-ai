## 1. Mục đích & cách dùng

NFRS đặc tả **chất lượng phải đạt** — không phải chức năng làm gì, mà chức năng phải chạy *nhanh thế nào, bền thế nào, an toàn thế nào*. Mỗi yêu cầu phi chức năng (NFR) đi kèm **Acceptance Criteria (AC)** dạng **Given / When / Then** đo được. AC ở đây là nền chung; AC chuyên biệt theo use case ở SRS 3.1 và UAT 3.6.

Phạm vi mục tiêu: aligned với Feature Tree v4.0 — 4 phase (P1 / P1.5 / P2 / P3). Target NFR thay đổi theo phase.

**Mức ưu tiên (MoSCoW):** M (Must) · S (Should) · C (Could) · W (Won't this phase).

## 5.bis Permission Claims (Capability-based granular permissions) ⭐ NEW

> **Phiên bản:** thêm trong NFRS v1.1 (2026-05-21) — formalize concept đã được code repo + FE spec v1.1 dùng nhưng chưa documented BA layer.

### 5.bis.1 Định nghĩa & lý do

Bên trên RBAC role (4 enterprise + 3 admin), Kaori dùng **Permission Claims** — capability-based permission attached vào JWT `permissions[]` claim. Mỗi claim grant access tới 1 capability cụ thể (vd `approve_refund`, `manage_dlq`).

**Tại sao cần claims thay vì chỉ role:**
- **Granular hơn role:** ví dụ trong CS dept, MANAGER có thể approve refund nhưng không nhất thiết approve workflow promotion. Role MANAGER thuần không phân biệt được.
- **Cross-functional permission:** vài capability (vd `view_audit_log`) cần cấp riêng cho compliance officer, không nên gán cứng vào role.
- **Time-limited grant:** claim có thể cấp tạm thời (vd `platform_admin` cho 24h trong incident response) mà không đổi role chính.
- **Cross-tenant scope (STU-01):** Studio Analyst cần access nhiều tenant; claim `view_mcp` + tenant scope list trong JWT giải quyết clean hơn role-based.

### 5.bis.2 Catalog claims chính thức

| Claim | Grants access to | Default for role/dept | Time-limited? |
|---|---|---|---|
| `view_audit_log` | P2-21 AI Decision Log standalone, P2-36 audit panel | None (cấp riêng) | No |
| `view_observability` | P1-13 Platform Health (SRE only) | SRE | No |
| `manage_dlq` | P5-04 DLQ Console | Platform Eng | No |
| `view_mcp` | P5-02 MCP Console (Studio analyst) | STU-01 (alias STUDIO_ANALYST in code) | No |
| `review_guardrails` | P5-01 Guardrails Review | SEC | No |
| `platform_admin` | All P1 screens | SUPER_ADMIN, ADMIN | Yes — incident response 24h max |
| `approve_workflow_promotion` | P2-27 Mode 2 "Promote B" button | MANAGER (auto-grant) | No |
| `triage_cs_tickets` ⭐ | P2-33, P2-34 | OPERATOR auto-grant if dept=CS | No |
| `approve_refund` ⭐ | P2-36 approve action | MANAGER auto-grant if dept=CS | No |
| `run_churn_save_action` ⭐ | P2-37 action panel | OPERATOR+ auto-grant if dept=CS | No |

> **Naming convention:** snake_case, verb-noun pattern (`verb_object` hoặc `view_resource`). Đăng ký claim mới qua CR Review Board.

### 5.bis.3 Auto-grant logic (BE auth-service)

Khi user login, BE auth-service compute `permissions[]` claim cho JWT:

```pseudo
function compute_permissions(user):
    claims = []
    # Role-based auto-grant
    if user.role == 'MANAGER':
        claims += ['approve_workflow_promotion']
    if user.role == 'SUPER_ADMIN' or 'ADMIN':
        claims += ['platform_admin']
    
    # Dept-based auto-grant
    if user.dept == 'CS':
        if user.role in ['OPERATOR','ANALYST','MANAGER']:
            claims += ['triage_cs_tickets', 'run_churn_save_action']
        if user.role == 'MANAGER':
            claims += ['approve_refund']
    
    # Studio (STU-01) cross-tenant
    if user.role == 'STU-01':
        claims += ['view_mcp']
    
    # Manual grants (from grants_phase2_tables mig 084)
    claims += user.manual_grants
    
    # Time-limited grants (with TTL check)
    for grant in user.temporary_grants:
        if grant.expires_at > now():
            claims += [grant.claim]
    
    return distinct(claims)
```

**AC mẫu NFR-SEC-15 (fail-closed):**
> *Given* endpoint `POST /cs/refunds/{id}/approve` declare `required_claim='approve_refund'`,
> *When* user không có claim trong JWT gọi endpoint,
> *Then* BE trả 403 + RFC 7807 Problem Details với `code='USR-ERR-403-CLAIM'` + audit event `claim_check_failed`; không expose chi tiết internal về claim definition.

**AC mẫu NFR-SEC-17 (TTL):**
> *Given* user được cấp `platform_admin` với expires_at = now+24h cho incident X,
> *When* sau 24h, JWT refresh hoặc gọi protected endpoint,
> *Then* claim removed; 403 nếu retry; audit `claim_expired_auto_revoked`.

### 5.bis.5 RBAC vs Claims — khi nào dùng cái nào

| Loại permission | Cách check |
|---|---|
| Hierarchy quyền chung (V/O/A/M) | RBAC role check `useRequireRole('MANAGER')` |
| Capability đặc thù (audit, refund, DLQ, etc.) | Claim check `useRequireClaim('approve_refund')` |
| Combo (role tối thiểu + capability cụ thể) | Cả 2 — claim check tự suy ra role nếu auto-grant có |

> Pattern khuyến nghị: **role cho UI navigation (hide nav items), claim cho action gates (hide CTA buttons)**.

## 13. Acceptance Criteria chung — Pattern bắt buộc

### 13.1 Pattern Given/When/Then bắt buộc cho mọi User Story
Mọi US trong URD phải có AC viết theo dạng:
```
Given <tiền điều kiện đo được>
When <hành động cụ thể>
Then <kết quả đo được + định lượng>
And <kết quả phụ nếu có>
```

### 13.2 Confidence-based Action Policy (gắn Insights Engine)
| Confidence + Calibration | Hành vi bắt buộc |
|---|---|
| conf ≥ 0.85 AND ECE < 0.05 | Hiển thị "TRUSTED" badge; action button nổi; auto-execute OK cho ENT MAX có policy |
| conf ≥ 0.85 AND ECE > 0.10 | Hiển thị "OVER-CONFIDENT WARNING"; bắt buộc manual verify |
| conf 0.65–0.84 | "REVIEW NEEDED"; bắt buộc verification step; show top-3 SHAP |
| conf 0.40–0.64 | "AI UNCERTAIN"; khuyến nghị manual investigation; show alternative scenarios |
| conf < 0.40 | Không hiển thị actionable; log "AI insufficient confidence"; trigger fallback hierarchy |

### 13.3 Negative scenario coverage bắt buộc
Mỗi US phải có AC cho 4 case tối thiểu:
1. **Happy path** — input hợp lệ → kết quả mong đợi
2. **Validation failed** — input sai → error message tiếng Việt cụ thể
3. **Permission denied** — user không đủ quyền → 403 + log
4. **Dependency unavailable** — service phụ thuộc down → graceful degradation (cache/fallback) hoặc 503 + Retry-After

- Test steps

- Pass/Fail criteria
