## 1. Giới thiệu

### 1.1 Mục đích
SRS trả lời **"Phần mềm phải làm gì, chính xác đến mức kiểm thử được"**. Đây là cầu nối Stakeholder Requirement → thiết kế & code. SRS này đặc biệt làm rõ: (a) **kiến trúc dữ liệu Medallion 3 lớp** (sửa thiếu sót lớp Silver/Bạc), (b) **happy case to-be**: workflow theo phòng ban + upload tài liệu + thực thi theo từng bước + xử lý 2 loại tài liệu (có cấu trúc / không cấu trúc).

### 1.2 Phạm vi
- **AS-IS:** chức năng đã có trong Feature Tree v4.0 (pipeline structured data, Process Mining, NOV, Adoption).
- **TO-BE (GAP):** Org/Department workflow + Document-into-workflow + step execution + xử lý unstructured. Các GAP-01/02/03 (FRD §3) được đặc tả ở đây để hiện thực hoá happy case; cần Change Request phê duyệt trước khi build.

### 1.3 Định nghĩa thuật ngữ
| Thuật ngữ | Định nghĩa |
|---|---|
| Medallion | Kiến trúc dữ liệu 3 lớp: Bronze (Đồng) → Silver (Bạc) → Gold (Vàng) |
| Workflow | Chuỗi step có nhánh (if/else/switch) gắn với 1 phòng ban; mỗi step = 1 card |
| Artifact | Sản phẩm trung gian truyền giữa các step (file thô, text đã trích, bảng đã chuẩn hoá, report) |
| Structured doc | Tài liệu **có cấu trúc**: CSV, Excel, JSON, TSV, export bảng DB — có hàng/cột, schema phát hiện được |
| Unstructured doc | Tài liệu **không cấu trúc**: PDF, Word, ảnh scan, hợp đồng, email, văn bản tự do — không có schema bảng |
| Document Intelligence Pipeline | Pipeline mới (GAP-03) xử lý unstructured: OCR/extract → classify → trích field → embed/RAG |

## 2. Kiến trúc dữ liệu Medallion — đặc tả đầy đủ 3 lớp

> Sửa thiếu sót đã nêu: trước đây Silver chỉ được liệt kê tên. Dưới đây định nghĩa **cả 3 lớp ngang nhau**, lớp Bạc nói rõ vai trò.

| Lớp | Mục đích | Đầu vào | **Biến đổi (transformation)** | Engine lưu trữ | Quality gate |
|---|---|---|---|---|---|
| **Bronze (Đồng) — RAW** | Source-of-truth, audit, replay | File/tài liệu thô từ upload hoặc connector | **KHÔNG transform.** Chỉ bọc envelope: tenant_id, source, ingest_ts, checksum SHA-256, raw_payload, lineage_tags | MinIO/S3 (Parquet + binary cho doc) | Ghi thành công + checksum khớp |
| **Silver (Bạc) — CLEANED & STANDARDIZED** | **Làm sạch + chuẩn hoá về schema chuẩn Kaori — interface contract nội bộ** | CDC/append từ Bronze + cleaning rules từ wizard | **(1)** Schema validation · **(2)** Type cast · **(3)** Null handling (drop/impute theo rule) · **(4)** Dedup (theo khoá nghiệp vụ) · **(5)** PII masking tiếng Việt (tên→role, SĐT→last4, email→hash) · **(6)** Normalize (SĐT→+84 E.164, tiền→VND, ngày→ISO 8601) · **(7)** Outlier flag · **(8)** Gắn lineage tag | ClickHouse (columnar), partition theo tenant + tháng | **≥80% records pass mới được lên Gold** |
| **Gold (Vàng) — BUSINESS-READY** | Sẵn sàng cho insight/report/NOV | Silver đã sạch | Aggregate · feature engineering · join đa nguồn · denormalize cho dashboard · populate Knowledge Graph | PostgreSQL MV + ClickHouse + Redis (hot cache) | Refresh thành công, view materialized |

**Nguyên tắc:** không có lớp nào bị bỏ qua. Dữ liệu **bắt buộc** đi Bronze → Silver → Gold. Bước Silver là nơi đảm bảo "dữ liệu sạch" — chính là giá trị khách thấy đầu tiên ("ồ data của tôi đã sạch").

## 4. Use Cases — TO-BE (HAPPY CASE) — cập nhật 2026-05-20 sang PARTIAL AS-IS

### UC-TB-01 — Tạo workflow công việc theo phòng ban (đơn vị nhỏ nhất)
- **Actor:** ENT-02 (Department Head), ENT-03
- **Tiền điều kiện:** workspace active; phòng ban đã tạo (AS-IS: danh sách phòng ban phẳng — GAP-01 nếu cần cây đa cấp tập đoàn)
- **Luồng chính:**
  1. Vào phòng ban (đơn vị tổ chức nhỏ nhất).
  2. "Tạo workflow mới" → mở canvas kéo-thả (**module 2.17 — Phase 2**).
  3. Thêm các step (card), nối bằng mũi tên; đặt nhánh điều kiện **if/else / switch**.
  4. Mỗi card khai báo: tên, mô tả, **note/chú thích**, **vùng tài liệu đính kèm** (GAP-02 — tổ chức dạng folder phân cấp), input artifact, action, output artifact, (tuỳ chọn) human-approval gate.
  5. Gán risk class (LOW/MED/HIGH/CRITICAL) → approval graph.
  6. Publish → ACTIVE (hoặc DRAFT do Process Mining sinh sẵn).
- **Hậu điều kiện:** workflow gắn đúng 1 phòng ban, versioned.
- **Quy tắc:** workflow là tài sản của phòng ban; phân quyền theo phòng ban + role; HIGH/CRITICAL phải qua approval + Blast Radius pre-check.
- **Trạng thái 2026-05-20:** **PARTIAL AS-IS** — BE workflow CRUD + step + folder + 45-node catalog + 25 production template + multi-user collab (editor/comment/lock) đã ship qua mig 053/054/068/069/072 (CLAUDE.md §14). FE drag-drop canvas đợi FE restructure; per-node permission matrix còn thiếu; UAT round 2 chưa làm.

### UC-TB-02 — Upload tài liệu vào workflow & thực thi theo từng bước
- **Actor:** ENT-04 (Operations User)
- **Tiền điều kiện:** workflow ACTIVE của phòng ban
- **Luồng chính:**
  1. Mở workflow → chọn step bắt đầu (card) → **upload tài liệu vào card** (dùng chức năng upload đã có; bổ sung gắn artifact vào node — GAP-02/03).
  2. Hệ thống **phân loại tài liệu**: *có cấu trúc* hay *không cấu trúc* (theo §5).
  3. **Thực thi theo từng bước:** mỗi step nhận `input artifact` → chạy `action` của step → sinh `output artifact` → truyền sang step kế (theo mũi tên). Nhánh if/else/switch định tuyến dựa trên: kết quả phân loại, độ tin cậy trích xuất, kết quả validate, hoặc giá trị nghiệp vụ.
  4. Step có human-approval gate → tạm dừng chờ duyệt, rồi tiếp.
  5. Step cuối sinh **báo cáo / kết quả / thống kê** theo phòng ban + ghi Decision Log + cập nhật NOV.
- **Hậu điều kiện:** mọi step có execution log (input/output/by/timestamp/rollback); báo cáo cuối lưu vào phòng ban.
- **Ngoại lệ:** step fail → DLQ + retry (Runtime Reliability); quá ngưỡng → rollback theo step + alert owner.
- **Trạng thái 2026-05-20:** **PARTIAL AS-IS** — Phase 2.7 đã hoàn thiện shape: 45/45 node executor + in-process runner + Temporal cron path + state machine (`WorkflowRunState`/`NodeRunState`) + event sourcing append-only + replay harness + saga compensation REVERSE walk + persistent idempotency ledger + 5-source DLQ console (CLAUDE.md §14 row Phase 2.6 + 2.7). ## 5. ĐẶC TẢ RÕ XỬ LÝ THEO 2 LOẠI TÀI LIỆU

> Đây là phần "cần nói rõ" theo yêu cầu. Mỗi loại nêu: nhận biết, đường đi qua Medallion, xử lý từng bước, đầu ra, và **hành vi mong muốn**.

### 5.1 Tài liệu CÓ CẤU TRÚC (Structured)

**Nhận biết:** đuôi `.csv .xlsx .xls .json .tsv`, export bảng DB; có hàng/cột; schema phát hiện được. Hệ thống auto-detect ở Bronze; nếu không chắc → hỏi người dùng xác nhận.

**Đường đi:** dùng lại pipeline Medallion sẵn có (AS-IS), gắn vào step workflow (GAP-02 wiring).

| Bước trong workflow | Lớp Medallion | Xử lý cụ thể | Output artifact |
|---|---|---|---|
| S1. Tiếp nhận | Bronze | Lưu raw + checksum + envelope; auto-detect encoding/delimiter/header | `bronze_table_id` |
| S2. Chuẩn hoá | **Silver** | Schema validate, type cast, null handling, dedup, **PII mask tiếng Việt**, normalize (SĐT/tiền/ngày), lineage; **gate ≥80% pass** | `silver_table_id` |
| S3. Phân tích | Gold | Aggregate + feature + join; chạy analysis template | `gold_table_id` |
| S4. Kết quả | Gold | Sinh báo cáo/thống kê + insight 3-tuyến + cập nhật NOV | `report_id`, `insight_id` |

**Hành vi mong muốn:**
- Step chỉ chuyển tiếp khi **Silver pass quality gate ≥80%**; nếu fail → dừng, hiện scorecard, đề xuất cleaning rule, chờ người dùng duyệt (không tự ý bỏ qua).
- Mọi PII được mask tại Silver **trước** khi tới Gold/insight hoặc external LLM.
- Kết quả cuối là số liệu định lượng (KPI, bảng, biểu đồ) gắn vào phòng ban + Decision Log.

### 5.2 Tài liệu KHÔNG CẤU TRÚC (Unstructured)

**Nhận biết:** `.pdf .docx .doc .jpg .png .txt`, email, hợp đồng, văn bản tự do; không có schema bảng. Cần **Document Intelligence Pipeline mới (GAP-03)** — đề xuất Phase 2–3.

**Đường đi:** Medallion mở rộng cho tài liệu.

| Bước trong workflow | Lớp / Thành phần | Xử lý cụ thể | Output artifact |
|---|---|---|---|
| U1. Tiếp nhận | Bronze (doc) | Lưu file gốc (binary) + checksum + metadata (loại, trang, nguồn) | `bronze_doc_id` |
| U2. Trích xuất | **Extraction (mới)** | OCR nếu ảnh/scan (OCR tiếng Việt); text+layout extraction nếu PDF/DOCX | `raw_text`, `layout` |
| U3. Phân loại & trích field | **Silver (doc)** | Classify loại (hợp đồng/hoá đơn/báo cáo/email…); LLM/IE trích **key-value + bảng + thực thể**; **PII detect+mask tiếng Việt**; chunking + embedding vào vector store; lineage | `silver_doc_record` (JSON field chuẩn) + `embeddings` |
| U4. Hợp nhất | Gold (doc) | Field đã trích → ghép schema chuẩn (có thể join với structured pipeline §5.1); index RAG-queryable | `gold_doc_view` |
| U5. Kết quả | Gold | Sinh báo cáo/tóm tắt/đối soát; Q&A RAG; trace compliance/audit; cập nhật NOV | `report_id`, `qa_index` |

**Hành vi mong muốn (cần rõ với khách):**
- **Bước nào không chắc → không bịa:** nếu độ tin cậy trích xuất < ngưỡng (vd 0.6) → step rẽ nhánh sang **human review card** (không auto pass).
- **OCR tiếng Việt** bắt buộc cho ảnh/scan; nếu OCR kém (điểm < ngưỡng) → cảnh báo, cho upload lại bản rõ hơn.
- Sau U3, tài liệu phi cấu trúc đã trở thành **bản ghi có cấu trúc** → từ đây tái sử dụng nguyên đường Silver→Gold như §5.1 (thống nhất 2 luồng tại Gold).
- RAG chỉ truy vấn trong phạm vi tenant + phòng ban (cô lập); trích dẫn nguồn (trang/đoạn) cho mọi câu trả lời.
- PII trong tài liệu phi cấu trúc phải mask trước khi gửi external LLM (BR-3).

### 5.3 Bảng so sánh nhanh 2 loại

| Tiêu chí | Có cấu trúc | Không cấu trúc |
|---|---|---|
| Ví dụ | CSV, Excel, JSON | PDF, Word, ảnh scan, email, hợp đồng |
| Pipeline | Medallion sẵn có (AS-IS) | **Document Intelligence mới (GAP-03)** |
| Bước đặc thù | — | OCR + extraction + classify + embedding |
| Điểm hợp nhất | — | Hội tụ với luồng structured tại **Gold** |
| Đầu ra | KPI, bảng, biểu đồ, insight | Field trích xuất + tóm tắt + Q&A RAG + báo cáo |
| Trạng thái | Phase 1 (pipeline) + GAP-02 (gắn vào card) | GAP-03 (toàn bộ) |
